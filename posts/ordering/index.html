2022-07-13

Say we are given

a≤mb≤mx≥my≥m\begin{aligned} a &\le m\\ b &\le m\\ x &\ge m\\ y &\ge m \end{aligned}abxy​≤m≤m≥m≥m​

and want to know, is x≥ax \ge ax≥a? How about x≥yx \ge yx≥y?

A human can of course look at these equations and say "yes" and "not provable from the given information", but our goal is to develop an algorithm that can do this for arbitrary sets of inequalities, and arbitrary queries. Our compiler will need this for some checks.

So, how can we do this? For starters, manipulating equations tends to be easiest if we place them into matrices. Secondarily, equalities are much easier to work with than inequalities. Thus, we introduce slack variables s∗≥0s_{*}\ge0s∗​≥0 to turn each of these into an equality. Now, we have

Now, say, we're given an arbitrary query with respect to our 5 variables (and possible constants). E.g., we may want to know whether a≤xa \le xa≤x or b+1≤yb + 1 \le yb+1≤y are known true.

Let us use a≤xa \le xa≤x, or equivalently, x−a≥0x - a \ge 0x−a≥0 as an example. We could repsent this as the column vector [−10010]T\begin{bmatrix}-1& 0&0&1&0\end{bmatrix}^T[−1​0​0​1​0​]T.

So, to be able to answer queries with respect to xxx and aaa, the first thing we'd need to be able to do isolate them in an expression. That means, we'll need to take some linear combination of the constraints to get the vector q=[−10010]T\textbf{q}=\begin{bmatrix}-1& 0&0&1&0\end{bmatrix}^Tq=[−1​0​0​1​0​]T.

But first, we should put our constraints into Hermite Normal Form and then drop all the zeroed constraints, or employ some other similar technique, in order to make sure that all remaining constraints are linearly independent. Guaranteeing they're linearly independent may simplify work later on. In the example above, we already have full row rank (4), so we'll skip this step in this blog post.

Now, to isolate our query, we have the problem

So we're trying to find a linear combination x\textbf{x}x of constraints to isolate our query q\textbf{q}q, but then what shall z\textbf{z}z be?

z\textbf{z}z corresponds to our four slack variables.

Now, to answer queries of the ≥\ge≥ variety, we want all slack variables to be ≤0\le0≤0. That is if we have w−s∗=dw - s_{*} = dw−s∗​=d, then w≥dw\ge dw≥d.

We can focus on only ≥\ge≥, as the other queries of interest can be reframed as ≥\ge≥, e.g. by adjusting constant offsets or flipping signs.

So, our current idea for proving (or failing to prove) a query is to find the linear combination of existing constraints that produces our query, and then check if all slack variables are negative.

Okay, but how do we solve for xxx? If A\textbf{A}A has full column rank, this is trivial: we either have a solution, or no solutions. If we have no solutions, we obviously fail. If we have one solution, it is trivial to check whether all xe≥0,e=E,…,2E−1x_e\ge0, e = E,\ldots,2E-1xe​≥0,e=E,…,2E−1.

But what if A\textbf{A}A does not have full column rank? Again, we have the problem Ax=q\textbf{Ax}=\textbf{q}Ax=q. Our first step will be to ensure that it has full row rank, which we can do by multiplying by some matrix U\textbf{U}U if necessary, e.g., calculated from Hermite Normal Form, discarding zeroed rows. Letting UA=H\textbf{UA}=\textbf{H}UA=H, and Uq=b\textbf{Uq}=\textbf{b}Uq=b, we now have Hx=b\textbf{Hx}=\textbf{b}Hx=b.

Letting MMM and NNN be the number of rows and columns of H\textbf{H}H, respectively. Note that MMM is also the rank of H\textbf{H}H, as we have guaranteed that it has full row rank. Because the column rank is deficient, we know N>MN > MN>M.

We can construct a non-singular (but not necessarilly unimodulary) N×NN\times NN×N integer matrix V\textbf{V}V, where the first MMM columns diagonalize H\textbf{H}H, while the remaining N−MN-MN−M columns are the nullspace of H\textbf{H}H. Calling these two blocks V1\textbf{V}_1V1​ and V2\textbf{V}_2V2​, we have V=[V1V2]\textbf{V} = \begin{bmatrix}\textbf{V}_1&\textbf{V}_2\end{bmatrix}V=[V1​​V2​​].

So, this reduces the problem to one of simply checking if Wy2≤c\textbf{Wy}_2 \le cWy2​≤c is satisfiable, where W\textbf{W}W is known from the initial constraints, and c\textbf{c}c is computed from the query (and most of the computation can be done once on initialization; for a particular query, all we need is a matrix-vector multiply). Checking feasibility is the first step to solving any linear program, but we can write specialized software for polyhedra that handles the unbounded nature of y2\textbf{y}_2y2​ without needing to duplicate the elements of W\textbf{W}W in a tableau.
